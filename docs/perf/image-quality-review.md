**Quality setting used (per-asset, not universal):**

| Asset | AVIF quality | WebP quality | Notes |
|---|---:|---:|---|
| `landing_mentors_20260607` | 70 | 85 | Mentor photo; faces require accurate skin tone |
| `landing_page` | 65 | 80 | Background / illustrative |

**Why per-asset (not a single universal q value):** The original plan
(`docs/plans/2026-06-12-001-…`) locked everyone onto a single quality
number, which is an anti-pattern per [web.dev / compress-images](https://web.dev/articles/compress-images).
A photo of a person needs q=65-70 to keep skin tones accurate, while an
illustration can drop to q=50-55 with no perceptible difference. Per-asset
review keeps both quality and size optimal.

**Manual visual review:** see `docs/perf-screenshots/` for side-by-side
comparisons. (Screenshots not yet captured; revisit once rendering happens
on physical low-end Android per the plan's validation protocol.)

**Re-generate variants:**

```bash
node scripts/build/optimize-images.js                  # all tracked assets
node scripts/build/optimize-images.js --asset=landing_mentors_20260607
```

The script is a dev/CI tool only. Variants are committed to the repo so the
runtime VPS (which intentionally has no `sharp` per AGENTS.md F-11) can
serve them as static files.
