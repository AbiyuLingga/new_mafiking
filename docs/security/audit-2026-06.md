# Mafiking security audit — June 2026

OWASP API Top 10 review and the broader Phase 1 application-layer audit for
Mafiking (`new_mafiking`). Source of truth for findings; each finding has
a status (Fixed / Documented / Tracked) and a severity (Critical / High /
Medium / Low / Info).

**Scope:** all HTTP routes (`server.js` + `routes/*`), the static-Babel
frontend in `src/*.jsx`, the dependency manifest (`package.json`), and the
runtime configuration. Excludes: deployment / WAF / TLS (Phase 4), AI
prompt-injection inventory (Phase 2), supply-chain CI (Phase 3).
**Status:** Review gate.

## Methodology

1. Built a route inventory (`docs/security/api-inventory.md`).
2. For every state-changing route, asserted that it returns 403 without
   a valid CSRF token (covered by `scripts/test-csrf-coverage.js`,
   25/25 pass).
3. Asserted that no CORS headers are echoed for arbitrary cross-origin
   requests (covered by `scripts/test-cors-regression.js`).
4. Swept `src/*.jsx` for `dangerouslySetInnerHTML`, `innerHTML`,
   `document.write`, `eval`, `new Function` (covered by
   `scripts/scan-xss-patterns.js`).
5. Reviewed all outbound HTTP calls in `routes/*` and `lib/*` for
   user-controlled URLs (SSRF).
6. Reviewed `vite.config.js` and `src/*.jsx` for accidental
   `process.env` / secret exposure.
7. Ran `npm audit` and captured the report.
8. Reviewed ownership check on every `:id` route and every cross-user
   query in `routes/progress.js`, `routes/correction.js`, `routes/payment.js`.

## Findings

### F-1 [High] Profile page XSS via unescaped `questionText` fallback
**Status:** Fixed in commit `<this commit>`.
**Location:** `src/profile.jsx:586-593` (and `src/generated-app.jsx:5632`).
**Description:** The "Latihan Bebas" header in the profile page used
`dangerouslySetInnerHTML` with a raw `att.questionText` fallback when
`window.renderMafikingMathHTML` was not loaded. The fallback path
returned the unescaped user-supplied text, which a self-XSS could turn
into an account takeover via a crafted attempt record.
**Remediation:** Use `window.escapeHtml` (now exported from
`src/practice.jsx`) as the fallback, then an inline `& < > " '` escape
if `escapeHtml` itself is missing.
**Regression guard:** `scripts/scan-xss-patterns.js` is wired into
`npm run check`; future regressions that re-introduce raw text in
`dangerouslySetInnerHTML` will fail the build.

### F-2 [High] Recommendation card XSS via unescaped `questionDisplay` fallback
**Status:** Fixed in commit `<this commit>`.
**Location:** `src/profile.jsx:160-164`
(`renderRecommendationQuestionHTML`).
**Description:** The recommendation card helper returned
`truncateRecommendationText(value)` raw when `window.renderMafikingMathHTML`
was not loaded. Same vector as F-1.
**Remediation:** Fall back to `window.escapeHtml` (or inline escape) so
the helper is safe even before the math library is ready.
**Note:** `renderRecommendationQuestionHTML` is added to the
`MANUALLY_REVIEWED_SAFE` allowlist in `scripts/scan-xss-patterns.js`
with a pointer to this audit entry.

### F-3 [Info] `escapeHtml` is not robust against Unicode / numeric character references
**Status:** Documented. Not exploitable today.
**Description:** `src/practice.jsx:2281` `escapeHtml` replaces `&`, `<`,
`>`, `"` but not `'` or backtick. Sufficient for the current HTML
contexts (double-quoted attributes and element bodies). Tracked in
`docs/security/posture.md` as a defense-in-depth improvement.

### F-4 [Info] `display_name` in profile is set via `xss()` in routes/auth.js
**Status:** Documented. No fix required.
**Description:** `xss()` from the npm `xss` library is applied to
`display_name`, `fakultas`, `username`, `phone_number`, etc. before
persistence. Output rendering is then either `textContent` (safe) or
goes through the math helper (safe). The combination is correct.

### F-5 [Info] Other `dangerouslySetInnerHTML` sites all use a safe helper
**Status:** Verified.
**Locations:**
- `src/practice.jsx:2555` `Eq` — uses `renderMafikingMathHTML`.
- `src/misi.jsx:418` `MissionQuestionText` — uses
  `renderMafikingMathHTML` with a safe `<span>{question}</span>`
  fallback when the helper is unavailable.
- `src/practice.jsx:2597` `MathNarrative` — uses
  `renderNarrativeHTML`, which itself uses `escapeHtml`.
- `src/practice.jsx:2555` (in `generated-app.jsx` mirror) — same.

The scanner confirms 8 hits total; all 8 pass through one of the
known-safe helpers (or the manually-reviewed-safe allowlist).

## BOLA scan (OWASP API Security #1)

Every `:id` route in the inventory was traced to the SQL or
storage call. Findings:

| Route | Filter | Verdict |
|---|---|---|
| `GET /api/quiz/chapters/:id/subtopics` | catalog read | no per-user claim |
| `GET /api/quiz/subtopics/:id/problems` | catalog read | no per-user claim |
| `GET /api/quiz/problems/:id` | catalog read | no per-user claim |
| `GET /api/quiz/subtopics/:id` | catalog read | no per-user claim |
| `GET /api/quiz/subtopics/:id/full` | catalog read | no per-user claim |
| `GET /api/tryouts/:tryoutId/full` | catalog read | no per-user claim |
| `GET /api/payment/status/:merchantOrderId` | `user_id = ? AND merchant_order_id = ?` | ownership check |
| `GET /api/correction/attempts` | `user_id = ?` | user-scoped |
| All `/api/admin/*` `:id` routes | `isAdmin` middleware | admin guard |
| All `/api/progress/*` state-changing | `req.session.userId` | user-scoped |

**Verdict:** No BOLA found. The pattern `WHERE user_id =
req.session.userId` is consistent across the user-scoped routes.
The `:id` ownership for catalog reads is implicit (catalog data
is public). Admin routes are gated by `isAdmin`.

## Sensitive data exposure (OWASP API Security #3)

- `src/*.jsx` contains no `process.env` references.
- `vite.config.js` does not use `define:` to inline any env values
  into the client bundle, so only `VITE_*` env vars are exposed to
  the browser (Vite default).
- `routes/auth.js:329` `/api/auth/me` returns the user's own row
  including `email`, `auth_provider`, and Clerk `clerk_id` — only
  the requesting user (session-bound) can see this.
- `routes/payment.js:560` `paymentGatewayState()` returns the
  public Duitku config to anyone, with a 30s cache. Verified that
  the secret key is not in the response.
- `routes/auth.js:404` `/api/config/clerk` returns only the
  publishable key. `CLERK_SECRET_KEY` is never read in this handler.
  Defense-in-depth: `scripts/test-cors-regression.js` asserts no
  secret-shaped string appears in the response body.

**Verdict:** No sensitive data exposure.

## CORS / browser hardening (OWASP API Security #8)

- Same-origin only. `scripts/test-cors-regression.js` confirms no
  `Access-Control-Allow-Origin` is set for any `/api/*` response,
  including for arbitrary `Origin` and `OPTIONS` preflights.
- `Content-Security-Policy-Report-Only` is on by default. The
  enforced (`Content-Security-Policy`) header is gated on a
  clean 7-day report window.
- `frame-ancestors 'none'` is in the CSP (always enforced, even in
  report-only mode) — clickjacking is blocked regardless of CSP
  mode.

## Resource consumption / DoS (OWASP API Security #4)

- `POST /api/auth/login` has a per-username 5-attempt / 5-minute
  lockout (`MAX_ATTEMPTS`, `LOCKOUT_MS` in `routes/auth.js`).
- `POST /api/auth/register` does **not** have a route-level rate
  limit. A scripted bot can keep creating new users, each of
  which auto-creates a SQLite row, an auto-guest session, and (on
  the first authenticated API call) a `user_progress` row. **This
  is a tracked Medium finding (F-6).**
- `POST /api/correction/transcribe` and `POST /api/correction/evaluate`
  are rate-limited via `express-rate-limit` (see `routes/correction.js`).
- `POST /api/performance/vitals` is rate-limited via
  `performanceLimiter`.
- `POST /api/csp-report` is not rate-limited but is no-op in the
  success path (204 + NDJSON append) and capped at 32 KB.

**F-6 [Medium] No rate limit on register / payment create.**
**Status:** Tracked. Fix planned for Phase 1.6.

## Function-level authorization (OWASP API Security #5)

- Admin routes use `isAdmin`. Verified for every route under
  `/api/admin/*`.
- Owner-only mutations (`/api/progress/submit`, `/api/correction/*`,
  `/api/payment/*`) all run behind `isAuthenticated` or
  `requireRegisteredUser`.
- The `localAdminMode` fallback in `middleware/admin.js` is
  scoped to loopback + `NODE_ENV !== 'production'` and only
  activates when `LOCAL_ADMIN_MODE=true`. This is a
  development-only escape hatch and is not exploitable in
  production.

## Security misconfiguration (OWASP API Security #8)

- `x-powered-by` is disabled (`server.js:322`).
- Helmet defaults are active (`server.js:323`): HSTS, `X-Content-Type-Options:
  nosniff`, `X-Frame-Options: SAMEORIGIN`, `Referrer-Policy: no-referrer`,
  etc.
- CORS not configured (same-origin only).
- `trust proxy` is `1` (server.js:321), appropriate for a single
  reverse-proxy setup (Cloudflare / Caddy).

## Improper inventory management (OWASP API Security #9)

- `docs/security/api-inventory.md` is the canonical route table,
  generated as part of this audit. Any new route file added to
  `routes/*.js` is auto-captured by re-running the inventory
  process. The `scripts/test-csrf-coverage.js` test currently
  enumerates the routes explicitly; if a route is added, it
  should be added to the test array (or the test should be
  rewritten to discover routes from the Express app).

**F-7 [Low] CSRF coverage test enumerates routes manually.**
**Status:** Documented. Acceptable for now; the test is the
explicit source of truth. A future improvement is to spin up the
real app and fuzz every `app.post|put|patch|delete` mount in
`server.js` + `routes/*.js`.

## Unsafe consumption of APIs (OWASP API Security #10)

- Duitku callback is verified by MD5 signature
  (`verifyCallbackSignature`).
- Clerk webhook is verified by svix
  (`routes/webhooks.js:handleClerkWebhook`).
- DeepSeek responses are stored verbatim in the question bank
  draft; an admin must review and commit before they reach users
  (`routes/admin-import.js`). This is the right trust boundary.
- Gemini image OCR / evaluation results are returned to the user
  who made the request; cross-user exposure is not possible
  because the result is not persisted without a user binding.
